# 用户管理

在项目首页点击左上角的 GNS3 Logo，选择"Management"（用户管理），进入管理页面。

<img style={{ width: '100%' }} alt="进入管理" src={useBaseUrl('img/web-ui/zh/mgmt-01-menu.jpeg')} />

Management 页面是 GNS3 Web UI 的管理中心，用于管理系统用户、权限和资源分配。通过这个页面，管理员可以控制谁能够访问 GNS3 系统、他们能执行哪些操作，以及如何分配计算资源。



## Users（用户管理）

创建和管理系统用户账户，支持认证、个人信息管理和组织关系维护。

<img style={{ width: '100%' }} alt="用户管理" src={useBaseUrl('img/web-ui/zh/mgmt-02-users.jpeg')} />



### 主要功能

**账户管理：**
- 创建用户账户，设置用户名、邮箱、全名和密码
- 编辑用户信息（用户名、邮箱、全名、密码）
- 删除用户账户（超级管理员除外）
- 查看所有用户列表和详情

**认证管理：**
- 用户登录认证（OAuth2 表单认证和 JSON 认证两种方式）
- 用户登出（撤销所有现有令牌）
- JWT 令牌机制，支持令牌版本控制
- 密码使用 bcrypt 哈希存储，不以明文保存

**个人信息管理：**
- 查看当前登录用户信息
- 更新个人信息（密码、邮箱、全名）
- 邮箱唯一性检查

**组织关系：**
- 查看用户所属的用户组
- 支持多用户组成员身份
- 组继承权限机制

### 账户属性

| 字段 | 说明 |
|------|------|
| username | 用户名，系统唯一 |
| email | 邮箱，系统唯一 |
| full_name | 用户全名 |
| is_active | 账户是否激活 |
| is_superadmin | 是否为超级管理员 |
| last_login | 最后登录时间 |
| token_version | 令牌版本（用于登出） |

### 安全限制

- 用户名：最少 3 个字符，只允许字母、数字、下划线和连字符
- 密码：最少 8 个字符，最多 100 个字符
- 用户名和邮箱在系统中必须唯一
- 超级管理员账户不能被删除

### 权限控制

用户管理操作需要特定权限：

| 特权 | 说明 |
|------|------|
| User.Audit | 查看用户列表和详情 |
| User.Allocate | 创建和删除用户 |
| User.Modify | 修改用户信息 |
| Group.Audit | 查看用户所属组 |

### 自动清理

删除用户时自动清理：
- 相关的 ACL 条目
- 用户与用户组的关联关系
- 使用数据库 CASCADE 机制确保数据一致性

### 使用场景

- 团队协作时，为每个成员创建独立账户
- 临时用户访问控制
- 审计和日志追踪

## Groups（用户组管理）


用户组是将多个用户组织成逻辑单元的机制，用于简化权限管理和用户组织。通过用户组，管理员可以一次性为多个用户分配相同的权限，避免逐个设置用户权限的繁琐工作。

<img style={{ width: '100%' }} alt="用户组管理" src={useBaseUrl('img/web-ui/zh/mgmt-03-groups.jpeg')} />



**系统特点：**
- 支持多对多关系：一个用户可以属于多个组，一个组可以包含多个用户
- 与 ACL 深度集成：通过组实现批量权限分配
- 内置组保护机制：系统预定义的组受保护不能删除
- 灵活的成员管理：支持动态添加和移除组成员

### 用户组组成要素

- **基本信息：** 组标识符（UUID）、组名称（唯一）、创建时间、更新时间、内置标识
- **成员列表：** 包含的所有用户账户，通过中间表建立多对多关系
- **权限关联：** 关联的 ACE，组内所有用户继承组的权限设置

### 主要功能

**用户组管理：**
- 创建自定义用户组，指定名称（必须唯一）
- 编辑用户组名称（内置组不可修改）
- 删除用户组（自动清理成员关联和 ACL 条目，内置组不可删除）
- 列出所有用户组及查看详细信息

**成员管理：**
- 添加现有用户到用户组（防止重复添加）
- 从用户组移除用户
- 查看用户组的完整成员列表

### 内置用户组

| 组名 | 说明 | 权限 |
|------|------|------|
| Administrators | 管理员组 | 与 Administrator 角色配合，拥有系统管理权限 |
| Users | 普通用户组 | 与 User 角色配合，拥有日常操作权限 |

### 命名规则

- 长度至少 3 个字符
- 只能包含字母、数字、下划线（_）和连字符（-）
- 系统中必须唯一
- 建议使用描述性名称，如 `network-team`、`project-admins`

### 权限控制

用户组管理相关特权：

| 特权 | 说明 |
|------|------|
| Group.Audit | 查看用户组和成员列表 |
| Group.Allocate | 创建和删除用户组 |
| Group.Modify | 修改用户组信息和管理成员 |

**权限继承：** 用户组成员自动继承组分配的所有权限。用户专属权限优先于组权限。

### 使用场景

- **按部门组织：** 网络部门组、系统管理部门组、开发部门组等
- **批量权限分配：** 为整个项目团队一次性分配访问权限
- **项目协作：** 不同项目组访问各自的项目资源
- **权限分级：** 管理员组、操作员组、只读访问组

### 重要注意事项

- 内置组不能被删除或修改基本信息
- 添加用户到组后立即继承组权限，移除后立即失去
- 删除用户组会清理相关 ACE 条目，不可逆
- 用户可能通过多个组获得权限叠加


## Roles（角色管理）


角色管理是 GNS3 权限控制系统的核心组件，通过定义用户权限级别来控制用户可以执行的操作类型。角色是权限的集合，每个角色包含多个特权，用户通过被分配角色来获得相应的操作权限。


<img style={{ width: '100%' }} alt="角色管理" src={useBaseUrl('img/web-ui/zh/mgmt-04-roles.jpeg')} />



**系统特点：**
- 基于特权的细粒度权限控制
- 支持内置角色和自定义角色
- 与访问控制列表（ACL）深度集成
- 提供职责分离和最小权限原则支持

### 角色组成要素

每个角色包含以下核心信息：

**基本信息：** 角色名称（唯一标识符）、描述、内置标识（内置角色受保护不能删除）

**权限集合：** 特权列表，每个特权代表一个具体操作权限，采用分层命名，如 `Project.Audit`、`Node.Allocate`

### 特权系统

GNS3 使用 46 个预定义特权来控制不同资源类型的操作：

| 权限类型 | 说明 | 示例 |
|---------|------|------|
| 查看权限 | 允许查看特定资源类型的信息 | Project.Audit、Node.Audit |
| 分配权限 | 允许创建和删除特定资源 | Project.Allocate、Node.Allocate |
| 修改权限 | 允许修改现有资源配置 | Project.Modify、Node.Modify |
| 特殊操作 | 特定功能的专门权限 | Node.Console、Link.Capture |

**支持的资源类型：** User、Group、Role、ACE、Project、Snapshot、Node、Link、Drawing、Symbol、Template、Image、Compute、Appliance、Pool

### 内置角色

GNS3 提供 7 个预定义的内置角色：

| 角色 | 特权数 | 主要功能 | 适用场景 |
|------|--------|---------|---------|
| Administrator | 46 | 拥有全部系统特权，跳过 ACL 检查 | 系统管理员 |
| User | 29 | 完整的项目管理、拓扑操作、设备控制台 | 网络工程师、普通操作人员 |
| Auditor | 11 | 所有资源的查看权限，不能创建/修改/删除 | 审计人员、监管人员 |
| Template Manager | 9 | 模板和镜像的创建、修改、查看 | 资源准备人员 |
| User Manager | 6 | 用户账户和用户组的创建、修改、删除 | 人力资源管理员 |
| ACL Manager | 6 | 角色和 ACE 的创建、修改、删除 | 安全管理员 |
| No Access | 0 | 不包含任何特权，用于明确拒绝访问 | 配合 ACL 的 DENY 规则 |

### 主要功能

- **创建角色**：创建自定义角色，指定名称和描述，名称必须唯一
- **编辑角色**：修改描述信息，添加或移除特权（内置角色名称和描述不可修改）
- **删除角色**：删除自定义角色，自动清理相关 ACL 条目（内置角色不可删除）
- **特权管理**：为角色分配或移除特权，同一特权可属于多个角色
- **查询功能**：列出所有角色，查看特定角色及其特权详情

### 角色保护机制

- 内置角色不能修改基本信息和删除
- 可以为内置角色添加或移除特权（不推荐）
- 删除角色时自动清理相关的 ACE 条目

### 使用场景

**权限分级管理：** 超级管理员（Administrator）→ 操作人员（User）→ 监管人员（Auditor）

**最小权限原则：** 为用户分配完成任务所需的最小权限集合，避免过度授权

**职责分离：**
- ACL Manager 负责权限配置
- User Manager 负责用户账户管理
- Template Manager 负责模板和镜像准备
- User 角色进行日常拓扑操作
- Auditor 角色进行合规检查

### 角色与权限集成

角色通过 ACE 将权限分配给用户或用户组，作为中间层简化权限管理。同一角色可以应用于多个路径和多个用户。

**权限检查流程：** 用户发起 API 请求 → 检查超级管理员 → 检查用户 ACE → 检查用户组 ACE → 验证角色特权 → 允许或拒绝

### 最佳实践

- 优先使用内置角色，确保系统稳定性
- 自定义角色遵循最小权限原则
- 严格控制 Administrator 角色的分配
- 使用 Auditor 角色满足查看需求
- 定期审查和清理不必要的自定义角色


## Pools（资源池管理）

资源池是项目的逻辑分组机制，用于将多个 GNS3 项目组织成便于管理的单元。每个资源池包含一组项目，可以对这些项目进行统一管理和访问控制。


<img style={{ width: '100%' }} alt="资源池管理" src={useBaseUrl('img/web-ui/zh/mgmt-05-pools.jpeg')} />



资源池主要用于项目组织，而不是计算节点资源（CPU、内存）的分配。计算资源由单独的计算节点管理。

### 主要功能

**资源池管理：**
- 创建资源池，为其指定唯一名称
- 编辑资源池名称
- 删除资源池（只删除组织关系，不删除实际项目）
- 查看所有资源池列表及详细信息

**项目管理：**
- 将项目添加到资源池
- 从资源池中移除项目
- 查看资源池中包含的所有项目

**权限控制：**
- 查看权限（Pool.Audit）：允许用户查看资源池和其中包含的项目
- 分配权限（Pool.Allocate）：允许创建和删除资源池
- 修改权限（Pool.Modify）：允许编辑资源池名称以及向池中添加或移除项目

### 使用场景

- **项目组织**：按部门、客户或课程分组项目
- **权限管理**：为整个资源池设置统一的访问权限
- **项目隔离**：不同团队使用不同的资源池，避免项目混淆

### 重要限制

- 一个项目只能属于一个资源池
- 目前只支持 GNS3 项目类型的资源
- 删除资源池不会删除实际项目文件或配置


## ACL（访问控制列表）

访问控制列表（ACL）提供基于 API 路径的细粒度访问控制，管理用户和组对不同系统资源的访问权限。ACL 由多个 ACE（访问控制条目）组成，每个 ACE 定义一条访问规则。

<img style={{ width: '100%' }} alt="访问控制" src={useBaseUrl('img/web-ui/zh/mgmt-06-acl.jpeg')} />



**系统特点：**
- 基于路径的权限控制，支持树形结构继承
- 与角色权限系统（RBAC）深度集成
- 支持用户级和组级权限设置
- 内置拒绝优先的安全机制

### ACE 组成要素

每个访问控制条目（ACE）包含以下关键要素：

**路径（Path）：**
- API 端点路径，定义规则应用的资源范围
- 支持从根路径到具体资源的所有层级
- 示例：`/projects`、`/projects/{project_id}`、`/templates`、`/pools/{pool_id}`
- 创建时系统会验证路径是否为有效的 API 端点

**对象类型：** 用户（user）或用户组（group）

**角色（Role）：**
- 关联到 ACE 的角色，定义具体的操作权限集合
- 角色包含多个特权，如 Project.Audit、Project.Allocate、Pool.Modify 等
- 必须在角色管理中预先定义角色和其包含的特权

**访问控制（Allowed）：**
- 允许（ALLOWED）：授予访问权限
- 拒绝（DENY）：明确拒绝访问，优先级高于允许规则

**传播（Propagate）：**
- 启用时：权限自动应用到当前路径下的所有子资源
- 禁用时：权限仅适用于当前精确路径
- 默认为启用状态

### 权限检查机制

**继承规则（优先级从高到低）：**

1. **超级管理员**：跳过所有 ACL 检查，拥有所有权限
2. **用户专属权限**：直接分配给用户的 ACE 规则
3. **用户组权限**：用户所属组的 ACE 规则
4. **路径深度优先**：深层路径的权限覆盖从上层继承的权限

**检查过程：**
- 系统从具体请求路径向根路径反向遍历
- 遇到 DENY 规则立即拒绝访问
- 遇到匹配的 ALLOWED 规则且传播条件满足则允许访问
- 用户权限总是优先于组权限进行检查

**DENY 优先机制：** 即使存在多个 ALLOWED 规则，只要有一个 DENY 规则匹配就会拒绝访问。

### 主要功能

- 创建新的访问控制规则（系统会验证路径有效性）
- 查看所有现有 ACE 列表
- 更新现有 ACE 配置
- 删除单个或批量删除 ACE
- 列出系统中所有可用于 ACL 设置的资源端点

### 支持的资源类型

| 分类 | 路径 | 说明 |
|------|------|------|
| 项目 | `/projects`、`/projects/{id}` | 所有项目及特定项目 |
| 项目子资源 | `/projects/{id}/nodes`、`/projects/{id}/links` | 节点和链路 |
| 模板 | `/templates`、`/templates/{id}` | 模板管理 |
| 镜像 | `/images`、`/images/{filename}` | 镜像管理 |
| 资源池 | `/pools`、`/pools/{id}` | 资源池管理 |
| 用户 | `/access/users`、`/access/users/{id}` | 用户管理 |
| 用户组 | `/access/groups`、`/access/groups/{id}` | 组管理 |
| 角色 | `/access/roles`、`/access/roles/{id}` | 角色管理 |
| ACE | `/access/acl`、`/access/acl/{id}` | ACL 管理 |

### 使用场景

- **项目隔离：** 为不同客户或部门的项目设置独立的访问权限

- **角色权限管理：** 管理员完全访问、操作员可创建修改、审计员只读

- **细粒度控制：** 用户可以创建项目但不能删除模板

- **安全防护：** 对敏感路径设置 DENY 规则防止误操作

### 典型配置示例

**限制用户组只读访问特定项目：**
- Path：`/projects/{project_id}`，类型：group，角色：Reader
- Allowed：是，传播：是

**完全禁止访问模板管理：**
- Path：`/templates`，类型：group，角色：任意
- Allowed：否，传播：是

**赋予用户创建项目权限：**
- Path：`/projects`，类型：user，角色：ProjectCreator
- Allowed：是，传播：否

### 重要注意事项

- 用户专属权限优先于用户组权限
- 深层路径权限覆盖上层继承的权限
- 权限检查实时进行，每次 API 请求都会验证
- 删除项目或资源池时自动清理相关 ACE
- 删除用户或组时，ACE 不会自动删除，需手动清理
- 创建 ACE 时路径必须对应系统中存在的 API 端点
- 超级管理员跳过所有 ACL 检查，不受任何 ACE 规则限制
- 所有 ACE 必须关联到已定义的角色
- 资源池内的项目有特殊的权限继承逻辑
